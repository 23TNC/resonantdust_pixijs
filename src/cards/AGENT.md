# AGENT.md

## Purpose
Runtime card objects. `DataManager` owns the card *data* (rows, columns); this module owns the card *objects* — the layout nodes that render and respond to input. Lives **scene-scoped**: `GameScene` creates a `CardManager` on enter and disposes it on exit, so map changes naturally rebuild the card population.

## Important files
- `Card.ts`: composite. Holds `(gameCard: GameCard, layoutCard: LayoutCard)`, subscribes to `data.subscribeKey("cards", cardId, …)` once, dispatches data changes to both halves. Static `create(cardId, ctx, cardManager)` factory reads the row + `definitions.shape(typeId)` to pick the right subclass pair. Detects zone changes and calls `cardManager.move`. Exposes `setDragging(value)` / `isDragging()` — forwards to both halves so game logic (overlap-push skip) and visual state stay in sync.
- `GameCard.ts`: abstract base for game-side state — position (decoded from row fields), drag state, slot membership. No rendering. Receives data via `applyData(row)`.
- `LayoutCard.ts`: abstract base for visual rendering, extends `LayoutNode`. Holds the Pixi container; receives data via `applyData(row)`. Self-attaches via `attach(zoneId)` which queries `ctx.layout.surfaceFor(zoneId)`; `detach()` removes from current parent. Card calls `attach` after initial data lands and re-attaches on zone change. Exposes `setHovered` / `setDragging` / `setSelected` / `setPending` boolean state flags — invalidate-on-change so subclasses redraw a cheap overlay without re-decoding the definition.
- `cardData.ts`: helpers for the position-related row fields (`microZone: u8`, `microLocation: u32`, `flags: u8`). `microZone` packs three sub-fields LSB-first: `stackedState:u2`, `localR:u3`, `localQ:u3`. Exposes `getStackedState(microZone)` / `clearStackedState(microZone)` / `setStackedState(microZone, newState)`, `decodeLooseXY(microLocation)`, `encodeLooseXY(x, y)`, plus `STACKED_*` constants. The position layout is dictated by the server schema; all consumers go through this module.
- `RectangleCard.ts`: `GameRectCard` (caches `flags` + `microLocation` from the row, exposes `getLoosePosition` / `setLoosePosition` writing the new `microLocation` via `data.cards.setClient`) + `LayoutRectCard` (rounded rect + name text + state overlay for hover/select/pending/drag). Uses `CardDefinition.style` colors when available; falls back to neutral colors + `"?"` name when the definition is missing. Caches `currentPackedDefinition` so applyData skips definition decode if unchanged. Self-positions from loose XY in the row.
- `HexagonCard.ts`: `GameHexCard` + `LayoutHexCard` skeletons — constructors warn, no logic/rendering yet.
- `CardManager.ts`: `Map<card_id, Card>` registry plus `Map<zoneId, Set<Card>>` per-zone index. Initial scan of `data.keys("cards")` plus `data.subscribeKeys("cards", …)` for adds/removes. Exposes `subscribe(zoneId, listener)` for per-zone Card delivery, `cardsInZone(zoneId)` for snapshot iteration, and `move(cardId, oldZoneId, newZoneId)` for re-routing on zone change. Does NOT parent LayoutCards — they self-attach to their zone's surface via `LayoutManager.surfaceFor`.

## Conventions
- **Boundary:** `CardManager` is dumb — it does NOT introspect row data. The factory in `Card.create` is the only place that reads from `DataManager` to pick a shape. CardManager just delegates and registers.
- **For card data, ask `DataManager`** (`ctx.data.get("cards", cardId)`). For the runtime UI object (position, drag state, the LayoutNode), ask `CardManager` (`cardManager.get(cardId)`).
- **Shape is immutable per `card_id`.** If a card's `packed_definition` ever updates such that its shape would flip (rect ↔ hex), the existing instance does NOT morph. We'd need to destroy + re-spawn — but in practice, server-side shape changes shouldn't happen for a given card.
- **Definition lookup is best-effort.** If `definitions.decode(packed_definition)` returns `undefined` (typo, missing definition), the card still renders with fallback visuals (`"?"` name, neutral colors) and continues to work — drag, hit-test, lifecycle all unaffected. Failure is visual-only.
- **Lifecycle is scene-scoped.** Created in `GameScene.onEnter`, disposed in `onExit`. On map change (different macro_zone), the scene tears down and rebuilds, naturally clearing the card population.
- **Cross-zone moves are handled by Card itself.** `Card.subscribeKey` fires on insert/update/delete for the row. `onDataChange` detects when `(macroZone, layer)` differs from `currentZoneId`, calls `layoutCard.detach()`, then `cardManager.move(cardId, oldZone, newZone)`, then `layoutCard.attach(newZone)`. Same Card instance preserved — game/render state survives. Subscribers in old zone hear `"removed"`; subscribers in new zone hear `"added"`.
- **Subscriptions go through `ctx.zones.ensure(zoneId)`**, not `spacetime.subscribeCards` and not `data.trackCards`. `GameScene` calls `ctx.zones.ensure(...)` and holds the release fn until `onExit`. DataManager listens to ZoneManager and drives the spacetime subscription.
- **Card UI is a view-filter on the data, not a 1:1 mirror.** Today CardManager spawns a Card for every `card_id` in `data.cards` (single-zone subscription, so this matches what's nearby). Eventually CardManager will hold a *tracked-zone set* (set of `ZoneId`): it only creates Cards for rows whose `packZoneId(row.macroZone, row.layer)` is in that set. Rows for other zones stay in `data.cards` but their UI representation is destroyed.

## Pitfalls
- The factory imports `RectangleCard` and `HexagonCard`, which both extend `Card`-side classes. The circular ESM imports work because the static method runs at call time, after all class declarations are evaluated. Don't move the factory into a constructor or static initializer that runs during module load.
- `Card` constructor calls `applyData` on game/layout halves synchronously when the row exists. Subclass `applyData` MUST be safe to run before subclass-specific fields are initialized — use property initializers, not constructor body, for any state read by `applyData`.
- `subscribeKey` listeners persist across delete. `Card.destroy` unsubscribes, but `CardManager.destroy(cardId)` — called on the keyset `removed` event — must be the path that disposes cards. Don't keep stale references to cards in other modules.
- `CardManager` is constructed AFTER `zones.ensure(...)` is kicked off. The initial scan via `data.keys("cards")` may find nothing if the spacetime subscription hasn't applied yet. That's fine — `subscribeKeys` will fire `"added"` for each row as it lands, and CardManager spawns then.
- When CardManager grows a tracked-zone set, the initial scan should switch from `data.keys("cards")` (everything) to `data.keysByIndex("cards", "zone", zoneId)` per tracked zone.
