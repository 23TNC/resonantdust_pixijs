# AGENT.md

## Purpose
Runtime card objects. `DataManager` owns the card *data* (rows, columns); this module owns the card *objects* — the layout nodes that render and respond to input. `CardManager` is **scene-scoped**: `GameScene` creates one on enter and disposes it on exit, so map changes naturally rebuild the card population.

## Important files
- `Card.ts`: composite. Holds `(gameCard: GameCard, layoutCard: LayoutCard)`, subscribes to `data.subscribeKey("cards", cardId, …)` once, dispatches data changes to both halves. Static `create(cardId, ctx, cardManager)` factory reads the row + `definitions.shape(typeId)` to pick the right subclass pair. Detects zone changes and calls `cardManager.move`. Also tracks `currentParentId` / `currentStackDirection` and the `stackedTop` / `stackedBottom` / `stackedHex` back-pointers (filled by sibling cards as they upsert their stacked state) so chain walks run in O(1) per step instead of scanning. Exposes `setDragging(value)` / `isDragging()` — forwards to both halves so game logic (overlap-push skip) and visual state stay in sync. The `setPosition(state)` API accepts loose/inventory/stacked/world position kinds, which is how drag-drop and recipe completion paths mutate cards.
- `GameCard.ts`: abstract base for game-side state — position (decoded from row fields), drag state, slot membership. No rendering. Receives data via `applyData(row)`.
- `LayoutCard.ts`: abstract base for visual rendering, extends `LayoutNode`. Holds the Pixi container; receives data via `applyData(row)`. Self-attaches via `attach(zoneId)` which queries `ctx.layout.surfaceFor(zoneId)`; `detach()` removes from current parent. Card calls `attach` after initial data lands and re-attaches on zone change. Exposes `setHovered` / `setDragging` / `setSelected` / `setPending` boolean state flags — invalidate-on-change so subclasses redraw a cheap overlay without re-decoding the definition.
- `cardData.ts`: helpers for the position-related row fields (`microZone: u8`, `microLocation: u32`, `flags: u8` — all top-level on `Card` rows). `microZone` packs three sub-fields LSB-first: `stackedState:u2`, `localR:u3`, `localQ:u3`. Exposes `getStackedState(microZone)` / `clearStackedState(microZone)` / `setStackedState(microZone, newState)`, `decodeLooseXY(microLocation)`, `encodeLooseXY(x, y)`, plus `STACKED_LOOSE` / `STACKED_ON_RECT_X` (top) / `STACKED_ON_RECT_Y` (bottom) / `STACKED_ON_HEX` constants. The bit layout matches the server's `magnetic.rs` `STACK_STATE_*` constants — the two must move together.
- `RectangleCard.ts`: `GameRectCard` (caches `flags`, `microZone`, `microLocation` from the row, exposes `getLoosePosition` / `setLoosePosition` writing the new `microLocation` via `data.cards.setClient`) + `LayoutRectCard` (rounded rect with title + state overlay for hover/select/pending/drag). Renders a `Sprite` whose texture comes from `ctx.textures.getRectTexture(definition, packed, titlePosition)` so all cards of one definition share a GPU texture region. Caches `currentPackedDefinition` so applyData skips re-fetching the texture if unchanged. Self-positions from loose XY in the row (or from its parent's stack-host when stacked).
- `HexagonCard.ts`: `GameHexCard` + `LayoutHexCard` — full implementation. `LayoutHexCard` renders a hex sprite from the texture atlas, exposes a `HexMount` child node (hit-passthrough surface for a rect-on-hex stacked card), runs the death-fade animation when the row's `dead` flag flips on, and (for cards bound to a magnetic action) drives the progress ring. Uses `world/worldCoords.ts` for hex grid math when placed on `WORLD_LAYER`.
- `RectCardVisual.ts` / `HexCardVisual.ts`: low-level reusable visuals (body fill, title, outline, name text). Used **only** by `TextureManager` to bake atlas slots — main-scene cards always render via `Sprite` reading the atlas, never these classes directly. Export `HEX_RADIUS` / `HEX_WIDTH` / `HEX_HEIGHT` / `RECT_CARD_WIDTH` / `RECT_CARD_HEIGHT` constants used across the world / inventory layout.
- `StatusBar.ts`: per-aspect emoji-icon table consumed by card overlays to render the aspect strip on a card's body. Edit the `ASPECT_ICONS` object to change rendered glyphs.
- `CardManager.ts`: `Map<card_id, Card>` registry plus `Map<zoneId, Set<Card>>` per-zone index. Initial scan of `data.keys("cards")` plus `data.subscribeKeys("cards", …)` for adds/removes. Exposes `subscribe(zoneId, listener)` for per-zone Card delivery, `cardsInZone(zoneId)` for snapshot iteration, `move(cardId, oldZoneId, newZoneId)` for re-routing on zone change, `subscribeStackChange(zoneId, listener)` consumed by `ActionManager` to know when a chain's topology shifts, and `stack(...)` / chain-walk helpers used by drag drops. Does NOT parent LayoutCards — they self-attach to their zone's surface via `LayoutManager.surfaceFor`.

## Conventions
- **Boundary:** `CardManager` is dumb — it does NOT introspect row data. The factory in `Card.create` is the only place that reads from `DataManager` to pick a shape. CardManager just delegates and registers.
- **For card data, ask `DataManager`** (`ctx.data.get("cards", cardId)`). For the runtime UI object (position, drag state, the LayoutNode), ask `CardManager` (`cardManager.get(cardId)`).
- **Shape is immutable per `card_id`.** If a card's `packed_definition` ever updates such that its shape would flip (rect ↔ hex), the existing instance does NOT morph. We'd need to destroy + re-spawn — but in practice, server-side shape changes shouldn't happen for a given card.
- **Card visuals come from the texture atlas.** Don't draw card bodies via `Graphics` on the main scene — use `ctx.textures.getRectTexture(...)` / `getHexTexture(...)` so every card of the same definition draws from one shared GPU texture. The state overlay (hover / drag / selected / pending) is a thin `Graphics` layered on top of the atlas sprite; that's the cheap, allowed path for per-card visual state.
- **Definition lookup is best-effort.** If `definitions.decode(packed_definition)` returns `undefined` (typo, missing definition), the card still renders with fallback visuals (`"?"` name, neutral colors) and continues to work — drag, hit-test, lifecycle all unaffected. Failure is visual-only.
- **Lifecycle is scene-scoped.** Created in `GameScene.onEnter`, disposed in `onExit`. On map change (different macro_zone), the scene tears down and rebuilds, naturally clearing the card population.
- **Cross-zone moves are handled by Card itself.** `Card.subscribeKey` fires on insert/update/delete for the row. `onDataChange` detects when `(macroZone, layer)` differs from `currentZoneId`, calls `layoutCard.detach()`, then `cardManager.move(cardId, oldZone, newZone)`, then `layoutCard.attach(newZone)`. Same Card instance preserved — game/render state survives. Subscribers in old zone hear `"removed"`; subscribers in new zone hear `"added"`.
- **Subscriptions go through `ctx.zones.ensure(zoneId)`**, not `spacetime.subscribeCards` and not `data.trackCards`. `GameScene` calls `ctx.zones.ensure(...)` and holds the release fn until `onExit`; world zones are reffed automatically by ZoneManager's anchor system. DataManager listens to ZoneManager and drives the spacetime subscription.
- **Stack state is server-authoritative.** Stacked-state bits in `microZone` and the parent card_id in `microLocation` are written by the server's `mirror_stack` (in `cards.rs`) and by the magnetic placement path. The client's drag-drop flow doesn't write these directly — it submits a stack via `submit_inventory_stacks` and the server echoes back the new row state.
- **Death is a flag, not a delete.** When a card is consumed by a recipe, the server sets `flags & FLAG_CARD_DEAD` (bit 7) and stamps `delta_t`, then schedules the actual row deletion ~10s later. `LayoutHexCard` / `LayoutRectCard` watch the bit and run a fade animation back-dated by `32 * delta_t` ms so late-subscribing clients see a sensible animation phase. Don't try to read "is this card dying" from the absence of the row — read the bit.

## Pitfalls
- The factory imports `RectangleCard` and `HexagonCard`, which both extend `Card`-side classes. The circular ESM imports work because the static method runs at call time, after all class declarations are evaluated. Don't move the factory into a constructor or static initializer that runs during module load.
- `Card` constructor calls `applyData` on game/layout halves synchronously when the row exists. Subclass `applyData` MUST be safe to run before subclass-specific fields are initialized — use property initializers, not constructor body, for any state read by `applyData`.
- `subscribeKey` listeners persist across delete. `Card.destroy` unsubscribes, but `CardManager.destroy(cardId)` — called on the keyset `removed` event — must be the path that disposes cards. Don't keep stale references to cards in other modules.
- `CardManager` is constructed AFTER `zones.ensure(...)` is kicked off. The initial scan via `data.keys("cards")` may find nothing if the spacetime subscription hasn't applied yet. That's fine — `subscribeKeys` will fire `"added"` for each row as it lands, and CardManager spawns then.
- The `stackedTop` / `stackedBottom` / `stackedHex` back-pointers are caches — authoritative truth lives in the children's row state. If two children claim the same parent slot via stale data, the last writer wins; the next refresh from the server reconciles. Don't iterate them without expecting brief inconsistency during a stack mutation.
- `cardData.ts`'s state constants (`STACKED_*`) must stay in lockstep with the server's `STACK_STATE_*` constants in `magnetic.rs`. Don't change one without the other — chain walks will silently misbehave.
